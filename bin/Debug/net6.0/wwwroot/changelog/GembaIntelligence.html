Gemba Intelligence Module

OEE data contains a wealth of unplanned downtime and reject detail. Manually examining this detail every Shift for each Asset is a time-consuming process and, even then, it is easy to miss small changes that can potentially become bigger issues in the future.

This is where Gemba Intelligence can help.

By using a data analysis technique called ‘anomaly detection’, values that that deviate from normal behaviour are automatically identified and raised as incidents that require further examination.

A suite of tools, including Gemba Intelligence Dashboard, Knowledge Base and Log Book Entry, are provided to help investigate and resolve any anomalies.

Anomaly Detection

Gemba Intelligence uses an algorithm to determine how far a particular data point is from the rest of the data. Technically speaking, the distance is known as the Z score and refers to the number of standard deviations (sigma or σ) away from the mean.

The frequency of values within a dataset gives a distribution similar to that shown below, where 68% of data falls within 1 standard deviation (i.e. with a Z score between -1 and +1), 95% within 2 standard deviations (Z score of -2 to +2) and 99.7% within three standard deviations (Z score of -3 to +3) from the mean.

The majority of data will have a Z score between 0 and +/-1. As Z score increases from the mean, the value becomes less common. A Z score of +/-3 or higher is commonly referred to as an outlier or an anomaly, but the actual Z score used can be configured within Gemba intelligence.

Using Availability % as an example, let’s assume the Availability of Machine 1 is commonly between 50% and 85% (the mean value 78%) over an 8-week period. The previous Shift Availability of 52% (Z score of 2.6) was within expected limits, but the current Shift Availability is 40.9% (Z score of 3.1). Clearly 40.9% is much lower than 78%, and so would be identified as an anomaly by Gemba Intelligence.

As another example, if Machine 2 typically runs between 75% and 78% then a Shift Availability of 70% would result in an anomaly being detected (Z score of 3.0). Even though this figure is above the Availability upper threshold of 60%, it is outside of the normal range and should be investigated. The key here is that the anomaly detection technique used by Gemba Intelligence adapts to the data.

Anomaly Types

The following values are assessed for anomalies each Shift:

  • Availability %
    • Total duration of every unplanned downtime
    • Total frequency of every unplanned downtime
    • Total MTTR (Mean Time to Repair) duration of every unplanned downtime
  • Performance %
  • Quality %
    • Total quantity of every reject definition

OEE % itself is not assessed since this is calculated from the Availability, Performance and Quality KPIs.

Configuring Gemba Intelligence for an Asset

In order to use Gemba Intelligence on an Asset the module must first be licenced and then the calculations enabled.

The Asset configuration grid in the Settings > Plant menu contains an Intelligence column to quickly show which Assets have Gemba Intelligence enabled.

To enable Gemba Intelligence for an Asset, Edit the Asset and navigate to the Gemba Intelligence tab. If the tab is not visible the module may not be licenced.

Click the Enabled switch to turn the feature on for that specific Asset, and pre-configure it with the default settings.

Calculation Window defines the time period in weeks used to calculate the mean and standard deviation used in the Z scores.

OEE Upper Threshold Filter is a behaviour which prevents an anomaly from being raised if the OEE\Availability\Performance\Quality % are above the respective upper thresholds for the Asset. For example, if the Availability upper threshold is 70% and the Shift Availability is 73% don’t raise an Availability anomaly. By default, this behaviour is disabled.

Planned Production Filter determines which Shifts are included in the calculations and eligible for anomaly detection. Only Shifts having Planned Production minutes greater than the filter minutes will be included. The default value is 0.

Net Operating Time Filter is similar to the Planned Production Filter but applies to the calculated figure of Total Units / Standard upm. Use this setting to ignore Shifts where the effect of the Total Units produced falls below the threshold. The default value is 0.

  • Availability Sensitivity sets the Z score threshold for Availability KPI anomalies. The default value is 3. The higher the value, the lower the sensitivity and less anomalies will be detected.
    • Duration Filter refers to unplanned downtimes and determines which Shift downtimes are included in the calculations and eligible for anomaly detection at an Event Reason level. The default value is 0.
    • Duration Sensitivity refers to unplanned downtimes and sets the Z score threshold for individual downtime detail anomalies. The default value is 3.
    • Frequency Filter refers to unplanned downtimes and determines which Shift downtimes are included in the calculations and eligible for anomaly detection at an Event Reason level based on the number of occurrences within each Shift. The default value is 0.
    • Frequency Sensitivity refers to unplanned downtimes and sets the Z score threshold for individual downtime frequency detail anomalies. The default value is 3.
    • MTTR Duration Filter refers to the mean time to repair of unplanned downtimes and determines which Shift downtimes are included in the calculations and eligible for anomaly detection at an Event Reason level. Mean Time to Repair is a function of duration and frequency. The default value is 0.
    • MTTR Sensitivity refers to the mean time to repair for unplanned downtimes and sets the Z score threshold for individual downtime MTTR detail anomalies. The default value is 3.
  • Performance Sensitivity sets the Z score threshold for Performance KPI anomalies. The default value is 3. The higher the value, the lower the sensitivity and less anomalies will be detected.
    • Asset Parameters allows the specific Parameters to be selected that are known to affect machine speed (Performance). Access to Parameter values is not common and the list will be blank if are no Parameters available.
  • Quality Sensitivity sets the Z score threshold for Quality KPI anomalies. The default value is 3. The higher the value, the lower the sensitivity and less anomalies will be detected.
    • Reject Quantity Filter determines which Shift rejects are included in the calculations and eligible for anomaly detection at a Reject Reason level. The default value is 0.
    • Reject Sensitivity sets the Z score threshold for individual reject detail anomalies. The default value is 3.

The Gemba Intelligence Asset configuration can be saved to multiple Assets, to reduce administration, through the use of the Copy Parameters button in the Asset configuration screen.

Anomaly Notifications and Status

When an anomaly is raised it is unacknowledged by default.

The Gemba Intelligence menu features a notification showing the number of unacknowledged anomalies within a configured timeframe for the Assets available to the current user.

The Map Overview also shows unacknowledged anomalies within the same timeframe by geographical location. Clicking View Anomalies opens the Gemba Intelligence Dashboard with corresponding filters applied.

The unacknowledged anomaly timeframe is configured in Settings > Account Settings > General and is a global value that can only be changed by a user with Account Administration rights. The default value is 1 day but this can be increased accordingly.

After an anomaly has been investigated or resolved there is the option to acknowledge it. This signals to other users that the anomaly is being dealt with and can also provide additional analysis opportunities.

Acknowledged anomalies are excluded from the Gemba Intelligence notifications and the Map Overview.

Use the Gemba Intelligence Dashboard to acknowledge anomalies either individually or in bulk.

Gemba Intelligence Dashboard

The dashboard is accessed from the main menu icon and consists of a filter panel and content area. The content area contains a default list view and a dashboard view.

List View

The list view displays anomaly summaries grouped by Asset, Shift and date that match the filter panel selections. The number of anomalies for each Asset / Shift is shown plus the OEE % and an approximate cost for the anomalies in both units and time when compared to the average (mean).

Clicking the header will sort the list so the lowest OEE or highest cost can be easily found.

Both the title and row checkboxes are used to acknowledge anomalies in bulk using the Acknowledge button in the top right of the screen. See below for more information on acknowledging an anomaly.

The chart icon next to the Asset name displays the OEE Anomaly Summary Chart, which provides a history of KPI level (Availability/Performance/Quality) anomalies on that Asset over the calculation window defined in the Asset configuration. Unacknowledged anomalies are shown as red circles and acknowledged anomalies are shown in grey. Clicking on the legend will show/hide values.

The chart icon next to the OEE figure opens the OEE Report using the same filters to provide additional detail.

Clicking the down arrow on the left of the Asset / Shift summary record displays the individual anomaly details which may be KPI or detail types.

Anomaly status is indicated by a red exclamation for unacknowledged and a green tick for acknowledged.

KPI anomalies refer to Availability/Performance/Quality values and these are identified by a highlighted row colour and the lack of Category, Code and Detail information. The Description shows why the anomaly was triggered, for example: “31.2% drop from the average of 88.9% to 57.8%”, and the approximate cost in both units and minutes when compared to the average (mean).

Note: KPI costs and detail costs are calculated independently. KPI anomaly costs are used when calculating the Shift summary row information with detail anomaly costs as a fall back.

Detail anomalies refer to unplanned downtime duration/frequency/MTTR or reject quantities and these will be displayed below their respective KPI anomalies, if they exist, with a white background. A KPI anomaly may not have any corresponding detail anomalies and vice versa. The Category, Code and Detail refer to a specific unplanned downtime Event Reason or reject. Here the description is tailored to suit the type. Minutes for unplanned downtime/MTTR and no units for frequency.

Anomaly Menu

The three vertical dots at the end of each anomaly row open a context menu containing various options.

Information menu item displays the Average (mean), the actual value and most importantly, the Z score. In this example, the Availability anomaly Z score relates to the Availability Z Score Sensitivity in the Asset configuration screen. Check the Z score and adjust the sensitivity, if required.

Acknowledge menu is used to acknowledge this specific anomaly, categorise it and provide optional details on why it occurred.

Acknowledged Anomalies are shown with a green tick instead of a red exclamation mark. Hovering over the icon displays the detail which is also available in Dynamic Reports.

Clicking the Acknowledge menu for an acknowledged anomaly gives the option to edit.

View Charts for Availability and Downtime/Frequency/MTTR anomalies displays an interactive pareto chart of unplanned downtimes for the Asset / Shift and a 14-day trend of both unplanned downtime duration and frequency.

When accessed via Availability, none of the individual downtimes are selected and the trend is total unplanned downtime and frequency for each Shift. Clicking on a pareto bar will highlight it and update the trend to show only the selected downtime.

The same chart is shown when accessed from a downtime duration/frequency anomaly but the corresponding pareto bar is pre-selected and the trend pre-filtered.

For rejects, the chart behaves in the same manner but displays a pareto of reject reasons instead.

View Charts option is not displayed for Performance anomalies.

Parameter Changes menu lists Asset parameter changes made +/- 24 hours of the anomaly. Not many Assets support the recording of parameter changes so this may be blank.

For Performance anomalies, the Asset parameter list is filtered by the Asset parameters selected in the Asset configuration screen. This allows users to focus on settings which primarily affect machine speed.

Log Book searches for any Log Book records for the Asset +/- 24 hours that contain information pertinent to the anomaly. It may be a component was replaced, a scheduled maintenance activity completed or even just a general comment relating to Asset function.

Knowledge Base is only available for detail anomalies and returns any Articles containing potential solutions for the corresponding unplanned downtime or reject reasons.

Create Action requires the Action module to be licenced and raises an Action to be assigned and resolved for the anomaly.

Actions are pre-populated with anomaly detail including the Asset and Shift.

Dashboard View

The Dashboard View is accessed by clicking the Dashboard button in the top right of the screen. The button changes to List View providing the method to return to the default screen layout.

The dashboard provides a summary of the anomalies selected in the filter panel showing total counts per type and daily counts per type. These can be useful when determining where to focus effort and identify patterns of increasing anomalies of a certain type.

Further analysis can be performed in Dynamic Reporting.

Dynamic Reporting

Gemba Intelligence reports have been added to Dynamic reporting to provide more advanced analysis than the Gemba Intelligence Dashboard.

Group anomaly types by Asset, Shift and even day of the week to gain further insights.

Summarize by unplanned downtime or reject reason anomalies to see worst offenders at site or machine level.

Raw data is available for export and includes acknowledge comments and categories.

Gemba Intelligence Email Subscription

The User Profile page contains Gemba Intelligence settings where a user can subscribe to receive email notifications when anomalies occur.

There is the option of:

  • Disabled
  • KPI level anomalies only (Availability/Performance/Quality)
  • KPI and detail which covers all anomalies
